# 開発ワークフロー / レビュー使い分け（always-on）

メインスレッドがオーケストレーター兼実装者。実装は**メインが直接** TDD で行う（実装専用エージェントは作らない）。
サブエージェントは「隔離・並列・専門化」の用途のみ。

## 標準フロー（issue 単位）

1. **issue 確認** — 受け入れ条件を把握する
2. **ブランチ** — `feat/issue-<番号>-<短い説明>`
3. **計画** — plan mode で調査 → 計画。引き継ぐ計画は `docs/plans/` に保存
4. **計画レビュー** — `@codex-plan-reviewer`（Codex MCP がある場合）。無ければ plan mode + ユーザー確認
5. **ユーザー承認**
6. **TDD 実装** — メインで Red → Green → Refactor。大量のテスト出力は `@tester` に隔離可
7. **レビュー** — 下表に従い `@code-reviewer`（＋専門レビュアー）を起動
8. **対応** — 指摘をメインで修正。再レビューは**指摘元のレビュアーのみ**再起動
9. **テスト** — `<test>` / `<lint>` / `<typecheck>` 通過
10. **PR 作成** → **CI 確認** → **マージ** → 次の issue

原因不明の不具合は `@debugger`（読み取り専用・原因分析のみ、修正はメイン）。

## レビュー使い分け

| 変更内容 | 起動するもの |
|---|---|
| 通常の実装・修正の差分 | `@code-reviewer` |
| 認証・認可・課金・入力検証・admin・API・middleware | `@code-reviewer` + `@security-reviewer`（並列・**security 必須**） |
| DB スキーマ・マイグレーション・SQL・クエリ | `@code-reviewer` + `@<db>-reviewer` |
| フロントエンド UI・画像・CSS・フォント | `@code-reviewer` + `@cwv-reviewer` |
| 構造変更・新規モジュール・抽象化境界 | `@code-reviewer` + `@solid-reviewer` |
| コメント・ドキュメント・設定値のみ | レビュー省略可 |

差分が複数領域にまたがる場合は該当レビュアーを**並列に発火**。レビュアーには「正確性に影響する指摘に絞る」と指示。

## 組み込みコマンド
- `/code-review` — 現在の diff を fresh サブエージェントでバグレビュー
- `/code-review ultra` — クラウド多段レビュー（PR 直前・課金あり・**人間が実行**）

## 禁止事項
- ユーザー承認前に実装を開始しない
- PR 前にレビュー（最低 `@code-reviewer`）をスキップしない
- セキュリティ／DB 差分で対応する専門レビュアーをスキップしない
- ユーザーの明示的な承認なしにコミット・プッシュしない
