# Issue 実装スキル（作業の型）

1つの Issue を小さく・レビュー可能で・テスト済みの差分で実装する**汎用の手順骨格**。
プロジェクト固有の手順（実行コマンド・必須レビュアー・ブランチ命名・PR テンプレ）は
各リポジトリの `CLAUDE.md` / `.claude/` に従って補う。

## Goal

Implement one issue with a small, reviewable, tested diff.

## Steps

1. Issue と受け入れ条件を読む。
2. 関連コードを調査する（既存パターン・再利用できる実装を探す）。
3. 計画を立てる（影響ファイル・リスク・必要なテストを特定）。
4. **実装前に計画をレビューする** — `plan-reviewer` で抜け・スコープ膨張・テスト不能性・前提誤りを検証する。**外部モデル（Codex 等）が使える環境ではそれを優先**する（例: プロジェクトの `codex-plan-reviewer`）。
5. 必要なら**ユーザー承認**を得る（非自明・挙動を大きく変える場合）。
6. テスト先行で、小さなステップで実装する。
7. プロジェクト固有のチェック（test/lint/typecheck/build 等）を実行する。
8. 差分をレビューする（code-reviewer ＋ 必要な専門レビュアー）。
9. 人間向けの要約（Human PR Review Brief）を用意する（`pr-review-brief` スキル）。

## Do Not

- スコープを広げない（依頼外の機能・無関係なリファクタを混ぜない）。
- ユーザー承認なしに commit / push / PR 作成をしない。
- 既存の公開挙動を、明示的に変える場合を除き壊さない。

## プロジェクト側で補うもの

- 実行する CI コマンド（実名）／必須レビュアーの起動条件
- ブランチ命名規則／PR 本文テンプレート
- ドメイン固有の確認項目
