# solid-review

## 概要

変更ファイルに対し、SOLID 原則と DRY / KISS / YAGNI の観点からレビューを行います。
2 エージェント構成（アクセル役 → ブレーキ役）で進めることで、原則違反の検出と過剰設計の抑制を両立します。

## 前提：レビュー対象の特定

引数で focus 指定や対象パスが渡されればそれに従います。指定がなければ、レビュー対象は以下の優先順位で決めます：

1. ステージ済み変更（`git diff --cached --name-only`）
2. 作業ツリーの変更（`git diff --name-only`）
3. 直近コミット（`git diff --name-only HEAD~1 HEAD`）

引数の例：
- `/solid-review` — 変更ファイル全体をレビュー
- `/solid-review focus on SRP violations` — SRP のみに絞る
- `/solid-review check src/services/ for DIP issues` — 対象パスと観点を絞る

## 実行フロー

### Step 0: プロジェクト文脈の収集（メインで実施）

Agent を呼び出す前に、以下を収集してエージェントに渡します：

1. **設計方針** — `CLAUDE.md` / `.claude/CLAUDE.md` / `.claude/rules/**/*.md` を Glob → Read します。設計方針セクションがあれば抜粋します
2. **技術スタック** — プロジェクトルートの `composer.json` / `package.json` / `Gemfile` / `pyproject.toml` / `go.mod` / `Cargo.toml` などを読み、言語・フレームワーク・主要依存を把握します
3. **アーキテクチャパターン** — Glob でディレクトリ構造を確認し、MVC / レイヤード / Clean Architecture / モジュラーモノリス 等を推定します
4. **変更ファイルリスト** — 上記「前提」で確定したファイル一覧です

これらをまとめた「プロジェクト文脈サマリ」を作成し、両 Agent への入力に含めます。

**設計方針が見つからなかった場合：** 構造と規模から妥当な方針を推定し、出力冒頭に推定前提を明示したうえで、レビュー末尾に次の案内を入れます：

> 💡 `CLAUDE.md` に設計方針を記載すると、より的確なレビューが可能になります。

### Step 1: Agent 1 — 構造レビュー（アクセル役）

`Agent` ツール（`subagent_type: general-purpose`）を使い、以下のプロンプトで起動します。プロジェクト文脈サマリと変更ファイル一覧を必ず同梱します。

```
あなたは構造レビュー担当（アクセル役）です。以下のプロジェクト文脈と変更ファイルを読み、
SOLID 原則 + DRY の観点で違反を検出してください。検出に集中し、過剰設計かどうかは判断しないでください。

【プロジェクト文脈】
<Step 0 で収集したサマリ>

【レビュー対象】
<変更ファイルパス一覧 + focus 指定>

【観点】
- SRP: 1 クラス/関数に複数のアクター（変更理由）が混在していないか。フレームワークのレイヤー規約
  （Controller / Service / Repository 等）に反してビジネスロジックが不適切な層に混入していないか。
- OCP: 種類追加のたびに既存コードの書き換えが必要な if-elseif / switch チェーンがないか。
  ※ 今後分岐が増える見込みがない場合は対象外。
- LSP: インターフェース/基底クラスを実装/継承したクラスが契約を破っていないか
  （黙って失敗、親にない前提条件の追加、戻り値の意味の変化など）。
- ISP: 実装クラスが使わないメソッドの実装を強制されていないか。
- DIP: 差し替えの需要がある境界で具体クラスへの直接依存（new のハードコード等）がないか。
  差し替え需要の有無はプロジェクト文脈から判断。
- DRY: 同一の知識（ビジネスルール、計算式、設定値）が複数箇所に重複していないか。
  ※ 変更理由が異なるコードの見た目の一致は重複とみなさない。

【各指摘に必ず含めること】
- 違反している原則名
- 該当箇所（ファイルパス:行番号）
- なぜ問題か（「呼び出し側は ○○ を前提にしている。しかしこの実装は △△ をしている。
  結果 □□ が起きる」の形式で具体的な実害を説明する）
- 修正方針の提案（プロジェクトの技術スタックに合った具体的な方法）

【出力形式】
番号付きリストで、各指摘を上記 4 項目を含む構造化フォーマットで返してください。
```

Agent 1 の出力（指摘リスト）を保持します。

### Step 2: Agent 2 — 過剰設計チェック（ブレーキ役）

別の `Agent`（`subagent_type: general-purpose`）を起動し、Agent 1 の指摘を引き渡して評価させます。

```
あなたは過剰設計チェック担当（ブレーキ役）です。Agent 1（アクセル役）が出した指摘リストを
一つずつ評価し、本当に「今」修正すべきかを判断してください。

【プロジェクト文脈】
<Step 0 で収集したサマリ>

【Agent 1 の指摘リスト】
<Step 1 の出力>

【評価観点】
- KISS: その修正はコードの複雑さを不必要に上げないか。
  初めてコードを読む人が 5 分以内に理解できる範囲に収まるか。
- YAGNI: その修正は「今」必要か。「将来必要になるかも」が唯一の根拠になっていないか。
- プロジェクト規模との整合性: CLAUDE.md の設計方針、ファイル数・チーム規模などの推定に照らして適切か。
- テスト環境の考慮: テストコードが存在しないプロジェクトでは、テスト容易性を主な根拠とした
  分離の指摘は優先度を下げる。

【判定区分】
- 🔴 即修正: 現時点で実害がある、または近い将来確実に問題になる
- 🟡 検討: 理論的に妥当だが、プロジェクト方針や規模次第で判断が分かれる
- ⚪ 見送り: 理論的には正しいが現時点では過剰設計

【出力形式】
指摘番号ごとに、評価コメント（なぜその判定にしたか）と判定マーク（🔴/🟡/⚪）を返してください。
```

### Step 3: 統合出力

メインで Agent 1 と Agent 2 の出力をマージし、ユーザーに次の形式で提示します：

```markdown
# solid-review レビュー結果

## 前提
- 技術スタック: <検出結果>
- アーキテクチャ: <推定>
- 設計方針の参照元: <CLAUDE.md / 推定>

## 指摘一覧

### 1. <原則名> — <ファイル:行>
**Agent 1（構造レビュー）**
<指摘内容と実害、修正方針>

**Agent 2（過剰設計チェック）**
<評価コメント>

**最終判定**: 🔴 即修正 / 🟡 検討 / ⚪ 見送り

---
（指摘の数だけ繰り返し）

## 🔴 修正アクションリスト
- [ ] <ファイル:行> — <要点>
- [ ] ...
```

🔴 が 1 件もない場合は、アクションリストの代わりに「即修正が必要な指摘はありません」と明示します。

設計方針が見つからなかった場合は、出力冒頭の「前提」直後に推定前提を明示し、末尾に CLAUDE.md 整備の案内を入れます。

## 注意事項

- 両 Agent はプロジェクト文脈を共有する必要があるため、Step 0 のサマリは必ず両方に渡します
- Agent 1 と Agent 2 は順序依存なので並列実行しません（Agent 2 は Agent 1 の出力を入力にします）
- 指摘がゼロの場合でも、レビュー対象ファイルと適用観点は必ず出力します（「何を見たか」を示すため）
- レビューはあくまで提案です。コードは変更しません。修正実行はユーザーの指示を待ちます
