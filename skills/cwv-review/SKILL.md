# CWV Review — Core Web Vitals観点のフロントエンドコードレビュー

## このスキルの目的

フロントエンドのコードやマークアップをCore Web Vitalsの3指標（LCP・INP・CLS）で体系的にレビューし、問題の検出と改善提案を行う。フレームワーク非依存で、AIの出力結果から既存コード、フレームワーク固有のコンポーネントまで幅広く対応する。

## レビューの進め方

### Step 1: 入力の確認

レビュー対象として以下のいずれかを受け取る。

- コードそのもの（HTML、JSX、Vue SFC、Blade、Svelteなど）
- AIに依頼したプロンプトとその出力結果
- Lighthouseスコアや計測結果と該当ページのコード
- 設計方針や構成の説明（コードがない場合）

コードがない場合は、問題の再現に必要な最小限の情報をユーザーに確認する。ただし過度なヒアリングは避け、得られた情報の範囲で最善のレビューを行う。

### Step 2: 3指標の体系的チェック

`references/checklist.md` を読み込み、チェックリストに沿って各指標ごとに問題を洗い出す。すべてが該当するわけではないので、対象コードに関係するものだけを報告する。問題がない指標は「問題なし」と明記して飛ばさず、ユーザーに安心感を与える。

レビュー結果は指標ごとにまとめ、各問題について「何が問題か」「なぜ問題か」「どう直すか」の3点を示す。

## レビュー結果の出力フォーマット

以下の構造で出力する。

```
## CWV レビュー結果

### 概要
（対象コードの要約と、全体的な評価を1〜2文で）

### LCP（表示速度）
- **問題なし** または 問題ごとに:
  - 問題: （何が起きているか）
  - 原因: （なぜCWVを悪化させるか）
  - 改善案: （具体的なコード例を含む修正方法）
  - 重要度: 高 / 中 / 低

### INP（操作の応答性）
- （同上の構造）

### CLS（視覚的安定性）
- （同上の構造）

### 総合評価
（最も優先度の高い改善から順に、対応すべきアクションをまとめる）
```

## レビュー時の判断基準

### 重要度の基準

- **高**: CWVの基準値を超える可能性が高い問題（LCP > 2.5秒、INP > 200ms、CLS > 0.1を招く構造的な問題）
- **中**: 基準値は超えないかもしれないが、ユーザー体験に影響する問題
- **低**: ベストプラクティスからの逸脱だが、実害は限定的な問題

### フレームワーク固有の知識の適用

コードからフレームワークを判別し、そのフレームワークが提供するCWV対策機能を踏まえてレビューする。

- **Next.js**: `<Image>`コンポーネントのpriority、`next/font`、Server Components、`useTransition`
- **Nuxt/Vue**: `<NuxtImg>`、`useAsyncData`のlazy、`<Suspense>`
- **Laravel Blade**: 素のHTMLとしてレビューし、`<link rel="preload">`やwidth/heightの明示的な指定を確認
- **Svelte/SvelteKit**: `enhanced:img`、コンポーネントの遅延読み込み
- **素のHTML**: ネイティブのloading属性、fetchpriority、preload/preconnectの活用

フレームワークが特定できない場合は、素のHTML/CSS/JSの標準的なベストプラクティスでレビューする。

### よくある誤検出を避ける

以下はCLSやパフォーマンスの問題として誤って指摘しやすいパターン。これらを問題として報告しないこと。

- 管理画面やログイン後の画面に対するSEO観点の指摘（CWVはUX指標としてのみ評価する）
- ファーストビュー外の画像へのpreload推奨（preloadはファーストビューのLCP要素にのみ使う）
- ファーストビューの画像へのloading="lazy"推奨（LCPを悪化させる。lazyはファーストビュー外の画像に使う）
- CSSのtransformプロパティによるアニメーション（レイアウトシフトを起こさない）
- ユーザー操作から500ms以内に発生するレイアウト変更（アコーディオン展開、タブ切り替えなど）
- 小さな要素の微小なシフト（CLS計算上インパクトが極めて小さい）
