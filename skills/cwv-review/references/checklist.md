# CWV レビュー チェックリスト

フロントエンドコードをレビューする際に、LCP・INP・CLSの各指標について確認する項目を網羅する。対象コードに該当する項目のみを拾って報告し、該当しない項目は無視する。

フレームワーク固有の改善手法は `framework-hints.md` を参照する。

## 目次

1. LCP（Largest Contentful Paint）チェック項目
2. INP（Interaction to Next Paint）チェック項目
3. CLS（Cumulative Layout Shift）チェック項目
4. 共通の誤検出パターン（指摘してはいけないもの）

---

## 1. LCP（Largest Contentful Paint）

**基準値**: 2.5秒以下が良好。4.0秒以上は要改善。
**主な対象**: ファーストビューで最も大きく表示される要素（通常はヒーロー画像、見出し、大きなテキストブロック）。

### 1.1 画像リソース

- [ ] LCP要素が画像の場合、`<img>` に `fetchpriority="high"` が付与されているか
- [ ] LCP画像に対して `loading="lazy"` が**付与されていない**か（付いていたらLCP悪化）
- [ ] LCP画像に対して `<link rel="preload" as="image">` が `<head>` に配置されているか
- [ ] 画像形式がWebP/AVIFなど最新フォーマットか、または`<picture>`でフォールバック指定されているか
- [ ] 画像の実寸より過度に大きいサイズを読み込んでいないか（`srcset`/`sizes`でレスポンシブ対応）
- [ ] ファーストビュー外の画像には `loading="lazy"` が付与されているか
- [ ] CDNまたはキャッシュ戦略が適切か（Cache-Controlヘッダ、長期キャッシュ）

### 1.2 フォント

- [ ] Webフォント読み込みに `<link rel="preload" as="font" crossorigin>` が使われているか
- [ ] `font-display: swap` または `optional` が指定されているか（`block`は避ける）
- [ ] フォントファイル形式が WOFF2 か
- [ ] 使用するsubsetやweightに絞り込まれているか（全weight読み込みはNG）

### 1.3 CSS / JavaScript

- [ ] クリティカルCSSがインライン化されているか、少なくともブロッキング時間が最小化されているか
- [ ] 非クリティカルCSSに `media` 属性や `onload` テクニックで非同期読み込みが適用されているか
- [ ] `<script>` に `defer` または `async` が付与されているか（ファーストビュー描画をブロックしないか）
- [ ] バンドルサイズが肥大化していないか（未使用import、tree-shaking漏れ、重複ライブラリ）
- [ ] Server Side Rendering / Static Generation が活用されているか（SPAで初期HTMLが空ならLCP悪化）

### 1.4 サーバー / ネットワーク

- [ ] 外部ドメインへの接続に `<link rel="preconnect">` または `dns-prefetch` が設定されているか
- [ ] TTFB（Time to First Byte）を悪化させる同期処理がサーバー側にないか
- [ ] レスポンスがgzip/brotli圧縮されているか（レビュー対象外の領域だが言及可）

## 2. INP（Interaction to Next Paint）

**基準値**: 200ms以下が良好。500ms以上は要改善。
**主な対象**: クリック・タップ・キー入力などユーザー操作からブラウザが次のフレームを描画するまでの時間。

### 2.1 重い同期処理

- [ ] イベントハンドラ内に同期的な重い計算（大量配列の`forEach`、正規表現、JSONパース等）がないか
- [ ] 重い処理がWeb Worker、`requestIdleCallback`、`setTimeout(0)` などでオフロードされているか
- [ ] ハンドラ冒頭で`preventDefault`等を呼ぶ前に重い処理が走っていないか

### 2.2 再レンダリング / 再計算

- [ ] 仮想DOM系フレームワークで不要な再レンダリングが発生していないか（親コンポーネントの再描画連鎖）
- [ ] 派生値がキャッシュされているか（React: `useMemo`、Vue: `computed`）
- [ ] イベントハンドラが毎レンダリングで再生成されていないか（React: `useCallback`）
- [ ] 大量リスト描画で仮想スクロール（windowing）が検討されているか
- [ ] 状態更新が緊急/非緊急で分離されているか（React: `useTransition`）

### 2.3 入力応答

- [ ] テキスト入力・検索ボックスの`input`イベントにデバウンス/スロットルが適用されているか
- [ ] クリック時にCSSフィードバック（`:active`、スピナー等）が即座に返るか
- [ ] サーバーラウンドトリップを伴う操作で「処理中」インジケータが即時表示されるか

### 2.4 サードパーティスクリプト

- [ ] 解析・広告・チャットウィジェット等のサードパーティJSがメインスレッドを占有していないか
- [ ] サードパーティJSが`async`/`defer`で読み込まれているか
- [ ] Long Task（50ms超のタスク）の発生源が特定されているか

## 3. CLS（Cumulative Layout Shift）

**基準値**: 0.1以下が良好。0.25以上は要改善。
**主な対象**: ページ表示後に予期せず発生するレイアウト変動の累積。

### 3.1 画像 / 埋め込み

- [ ] すべての`<img>`に`width`/`height`属性が指定されているか（またはCSSで`aspect-ratio`指定）
- [ ] `<iframe>`に`width`/`height`が指定されているか
- [ ] `<video>`に`width`/`height`または`aspect-ratio`が指定されているか
- [ ] 動的に挿入される画像の領域が事前に`min-height`等で確保されているか

### 3.2 フォント

- [ ] Webフォント読み込み完了時のFOUT/FOITでレイアウトが崩れていないか
- [ ] フォールバックフォントのメトリクス調整（`size-adjust`、`ascent-override`等）が検討されているか
- [ ] `font-display: swap` でもサイズが大きく変わるフォントは要注意

### 3.3 動的コンテンツ

- [ ] 広告、バナー、SNS埋め込みの領域が`min-height`等で確保されているか
- [ ] 非同期で読み込まれるコンテンツに対してスケルトンUIやプレースホルダが用意されているか
- [ ] `v-if` / 条件レンダリングで表示切り替えされる要素がレイアウトを押し下げていないか
- [ ] 上部に後から挿入される要素（通知バー、同意バナー等）がないか

### 3.4 CSS アニメーション

- [ ] アニメーションが `transform` / `opacity` で実装されているか（`top`/`left`/`width`/`height`はCLS発生）
- [ ] `position: absolute` / `fixed` で通常フローから切り離された要素は後続のレイアウトに影響しないか

---

## 4. 共通の誤検出パターン（指摘してはいけないもの）

以下は一見問題に見えるが、CWV上は実害がないか、むしろ推奨を誤解しているパターン。レビューで**指摘しない**こと。

- **管理画面やログイン後のSEO指摘**: CWVはUX指標であり、SEO文脈の指摘（meta、OGP、構造化データ等）は対象外。
- **ファーストビュー外画像への`preload`推奨**: `preload`はLCP要素のみに使う。それ以外にpreloadすると他リソースの帯域を奪いLCPを悪化させる。
- **ファーストビュー画像への`loading="lazy"`推奨**: LCPを悪化させる。`lazy`はファーストビュー外にのみ使う。
- **`transform`/`opacity`アニメーションへのCLS指摘**: これらはコンポジットレイヤーで処理されレイアウトシフトを起こさない。
- **ユーザー操作から500ms以内のレイアウト変更**: アコーディオン展開、タブ切替、モーダル表示などの意図的な変化はCLS計算から除外される。
- **小さな要素の微小シフト**: CLSスコアは `impact fraction × distance fraction` で計算され、画面の大部分を占めない要素の小さな動きはスコアに寄与しない。
- **初期描画前のレイアウト変更**: 描画前（First Paint前）のレイアウト確定はCLSにカウントされない。
- **過剰な`memo`適用**: React等で全コンポーネントに`React.memo`を適用するのは過剰最適化で、propsの比較コストで逆にINPを悪化させうる。
